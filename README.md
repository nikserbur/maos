# MAOS — Monitoring & Optimization System

Система мониторинга и **оптимизации производства** под неопределённостью.
В основе — **двухстадийная стохастическая модель**: сначала оптимизируется
**исполнение** плана (как произвести), затем **программа** (что производить).
Демо — на примере металлургического комбината полного цикла.

> **Не демо 3D-сцены, а работающая система:** все расчёты идут на настоящем
> **C++-бэкенде** из связанной БД (SQLite), а не на моках.

**Версия 0.18.0.** Версия пишется в `package.json`, статус-бар, окно входа и
`/api/health`. Статус стадий — в [docs/ROADMAP.md](docs/ROADMAP.md).

## Главная идея — две стадии, обе под неопределённостью

```
СТАДИЯ 1 — КАК произвести (исполнение):
  программа + 3D-граф оборудования + техкарты + рабочие + календари
  → поиск оптимального ПОРЯДКА операций (симуляция-оценщик, диспетч-правила)
  → диаграмма Ганта, загрузка оборудования, планы рабочих, узкие места, простои
  → длительности операций ТЯЖЕЛОХВОСТНЫЕ (Монте-Карло) ⇒ риск срыва срока

СТАДИЯ 2 — ЧТО производить (портфель):
  себестоимость/срок из Стадии 1 + ЦЕНЫ сбыта (стохастика, корреляция)
  → устойчивый ПОРТФЕЛЬ продукции (CVaR / worst-case / mean-variance / min-regret)
  → диверсификация под риск цен («не ставить всё на одно»)

СВЯЗЬ: портфель Стадии 2 → программа Стадии 1.
```

Цель — найти **не самое прибыльное, а самое устойчивое** решение к внешним
условиям. Полная модель: [docs/SCHEDULING.md](docs/SCHEDULING.md) (Стадия 1),
[docs/OPTIMIZATION.md](docs/OPTIMIZATION.md) (Стадия 2),
[docs/CONCEPT.md](docs/CONCEPT.md), [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Внешние условия и прогноз во времени (единый движок)

Цены и ключевая ставка — **внешние условия**, прогнозируемые **одним** стохастическим
движком (канон по разбору эталонных методик):

```
история ряда (ts_data / macro_ts)
  → μ из лог-доходностей, σ из реализованной волатильности
  → ПОДБОР распределения шага по AIC: нормаль ↔ Лаплас ↔ t-Стьюдент (тяж. хвосты)
  → Монте-Карло с общим рыночным фактором (корреляция) под макрофакторами
     (инфляция-дрейф, курс, индекс спроса)
  → веер P10/P50/P90 по месяцам
```

Так считаются **и цены изделий/сырья, и ключевая ставка**. Ставка — внешнее условие
верхнего уровня (через инфляцию задаёт дрейф цен). Режим **детерминированный**
схлопывает веер в точечный прогноз. Подробнее — [docs/ROADMAP.md](docs/ROADMAP.md)
(Стадия E+/E++, привязка к модулям эталонов).

## Технологический стек

- **Фронтенд:** React 18 + TypeScript + Vite 5; three.js / @react-three-fiber
  (3D-схема); дизайн-система Palantir Blueprint (dark-first), токены `src/styles/`.
- **Бэкенд:** **C++17** — `cpp-httplib` (локальный HTTP на `127.0.0.1:8080`),
  **SQLite**, `nlohmann/json`. Планировщик (`Scheduler.*`) и оптимизатор
  (`Optimizer.*`) — алгоритмическое ядро.
- Связь фронтенд ↔ бэкенд — JSON по `127.0.0.1`. Всё локально (local-first),
  без внешних сервисов в рантайме.

## Экраны (рельса навигации)

| Экран | Что |
|---|---|
| **Схема** | 3D-граф предприятия из БД (оборудование = узлы, связи = потоки); drill-down |
| **Справочники** | реестры НСИ, связанные по ID: типы оборуд., оборудование, изделия, техкарты, операции, рабочие |
| **План** | **Стадия 1**: программа заказов, MRP-материалы, рабочий календарь, диаграмма Ганта, загрузка/простои, планы рабочих, узкие места, сравнение правил |
| **Оптимизация** | **Стадия 2**: устойчивый портфель vs наивный максимум прибыли, распределение прибыли, вклад в риск |
| **Сценарии** | внешние условия: распределения цен, корреляция ρ, β по изделиям |
| **Сравнение сценариев** | единый сценарий (цены + цель оптимизации + режим); клонирование; прогон двух сценариев **бок о бок** с подсветкой лучшего |
| **Прогноз цен** | прогноз цен изделий/сырья и **ключевой ставки** во времени (веер P10/P50/P90), макрофакторы (инфляция/курс/спрос), подобранное распределение |
| **Администрирование** | журнал действий (Action layer), пользователи, роли/права (RBAC) |

## Что уже реализовано

- ✅ **Связность реестров по ID** (FOREIGN KEY + join-таблицы) — себестоимость и
  мощности считаются по производственной цепочке (`operation_inputs`, BOM).
- ✅ **3D-схема хранится в БД** (узлы/связи/мета), `GET /api/scheme`.
- ✅ **Стадия 1** — FJSP-планировщик: развёртка программы в граф операций,
  списочное планирование, **тяжелохвостные** длительности, Ганта/загрузка/планы
  рабочих/узкие места/простои, KPI (makespan/tardiness/OTD), сравнение диспетч-правил.
- ✅ **Стадия 2** — устойчивая оптимизация портфеля под стохастику **коррелированных**
  цен; диверсификация; вклад изделий в риск.
- ✅ **Связь стадий** (портфель → программа).
- ✅ **Стадия A** — производственная программа (заказы).
- ✅ **Стадия B** — расписания и календари (операции «растягиваются» через смены/выходные).
- ✅ **Стадия C** — запасы / MRP (разузлование программы → потребность в материалах,
  дефицит/дозаказ).
- ✅ **Стадия D** — Action layer + экран «Администрирование».
- ✅ **Стадия E+** — внешние условия и динамика во времени: прогноз цен по месяцам
  (веер), макрофакторы (инфляция/курс/спрос), **загрузка оборудования во времени**
  (тепловая карта станок×время из Ганта).
- ✅ **Стадия F0/F1** — историческая БД цен `ts_data`/`macro_ts`; прогноз **от истории**
  (μ/σ из лог-доходностей); **подбор распределения по AIC** (нормаль/Лаплас/t-Стьюдент);
  **единый движок** для цен и ключевой ставки.
- ✅ **Стадия E** (ядро) — менеджер сценариев: сценарий = цены + **цель оптимизации**
  + макропараметры прогноза + **режим** детерм./стох.; **клонирование**; **сравнение
  бок о бок**; один сценарий гоняет и прогноз, и оптимизацию.
- ✅ **3D-территория**: цеха-здания (домна/сталеплавильный/прокатный/энерго/склад),
  ортогональная дорожная сеть с развязками и транспортом, интерьер цеха (пол, кран,
  погрузчики), декор (КПП, шлагбаум, парковка, деревья); **универсальная вложенность**
  (узел с потомками = здание на любом уровне).
- ✅ Вход через реестр пользователей (RBAC), миграции БД с сохранением данных.

Дальнейшее (E — `scenario_override` и привязка к плану, F1 — Парето/α-stable,
F — мониторинг/KPI, G — окно входа + RBAC-энфорсмент, H — импорт/экспорт,
I — перепланирование, J — метаэвристики, K — обратная связь 1→2, L — MENTAT) —
в [docs/ROADMAP.md](docs/ROADMAP.md).

## Установка и запуск

Нужно поднять **два** процесса: C++-бэкенд и веб-фронтенд.

### 1. Бэкенд (C++)

```bash
cd backend && mkdir -p build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
cmake --build . -j
./maos_server maos.db        # http://127.0.0.1:8080
```

При первом запуске на пустой БД данные **посеются автоматически** (демо-комбинат:
6 цехов с оборудованием внутри, изделия, техкарты, операции, рабочие, заказы,
сценарии цен, история цен/ключевой ставки). Миграции (до v12) идемпотентны и
**сохраняют существующие данные**.

### 2. Фронтенд (веб)

```bash
npm install
npm run dev                  # http://localhost:5173 (проксирует /api на :8080)
```

Открыть **http://localhost:5173**, вход: **`admin` / `maos2025`**.

## Команды

| Команда | Назначение |
|---|---|
| `npm run dev` | дев-сервер фронта с HMR |
| `npm run build` | проверка типов + продакшн-сборка в `dist/` |
| `npm run typecheck` | только проверка типов |
| `cmake --build backend/build -j` | сборка бэкенда |
| `./backend/build/maos_server maos.db` | запуск бэкенда |
