# MAOS — Стадия 1: оптимизация исполнения плана (как произвести)

> Каноническое описание **первой стадии** двухстадийной стохастической модели
> (см. [docs/ARCHITECTURE.md](ARCHITECTURE.md) ADR-003 и внешний ТЗ «Двухстадийная
> стохастическая модель», «Требования к производственному плану/оптимизации/
> эвристики», «Время выполнения операции»). Реализация — `backend/src/Scheduler.*`,
> экран «План». Стадия 2 (что производить) — [docs/OPTIMIZATION.md](OPTIMIZATION.md).

## 0. Идея

**Дано:** производственная программа (что и сколько, к сроку) + 3D-граф
оборудования (типы, связи) + техкарты (последовательности операций) + рабочие +
календари. **Найти:** оптимальный **ПОРЯДОК** операций, который симуляция
разворачивает в расписание (Ганта, загрузка, планы рабочих), с минимумом целевой
функции `F = w_time·ВРЕМЯ + w_cost·СТОИМОСТЬ + w_risk·РИСК`. Длительности операций —
**тяжелохвостные** (редкие длинные задержки доминируют в риске срыва срока).

```
порядок операций (решение) ──► СИМУЛЯЦИЯ (оценщик) ──► расписание + KPI(время/стоимость/риск)
       ▲                                                          │
       └──────────── поиск лучшего порядка (диспетч-правила) ◄─────┘
```

## 1. Развёртка программы в граф операций

Каждый заказ `(изделие, кол-во, срок)` разворачивается в **операции по всей
производственной цепочке**:

- операции техкарты изделия в порядке `order_no` — предшествование внутри маршрута;
- **входы операций** (`operation_inputs`, по ID) тянут операции изготовления
  полуфабрикатов: операция-потребитель не стартует раньше последней операции
  производства её входного изделия (межмаршрутное предшествование);
- общий полуфабрикат питает много изделий — **DAG** (защита от циклов);
- объём заказа масштабирует длительность (`sizeFactor`), наладка `setup_time`.

## 2. Симуляция (списочное планирование, активное расписание)

Идём по операциям в порядке приоритета; каждую ставим на **самый ранний**
допустимый момент при ограничениях:

1. предшествование (маршрут + входы);
2. станок — одна операция в момент; внутри **типа оборудования**
   (`operation_wc_types`) выбираем наименее загруженный станок;
3. **рабочий** — dual-resource (наименее загруженный из доступных);
4. (расширяемо) календарь/смены, наладки, материальная доступность, 3D-связи графа.

Выход одного прогона: для каждой операции `(станок, рабочий, старт, финиш)` и
makespan/просрочка.

## 3. Тяжелохвостные длительности (стохастика Стадии 1)

`t_exp = (min+4·likely+max)/6`, `σ = (max−min)/6` (PERT) → **логнормаль** (тот же
средний/σ) + **примесь Парето**: с вероятностью `tail_weight` длительность
умножается на тяжёлый множитель (`tail_index` — индекс хвоста), моделируя
поломку/переделку/ожидание; брак (`risk_coef`) добавляет переделку. Параметры — на
операцию (`operations.tail_weight/tail_index`).

Метод Монте-Карло (N прогонов) даёт **распределение** makespan/просрочки:
- `makespan_mean` (истинное среднее > детерминированного на номинале — эффект хвоста);
- `makespan_cvar` (среднее худших α — хвостовой риск), `makespan_worst`;
- `tardiness`, `tardiness_cvar`, `OTD` (доля заказов в срок).

## 4. Оптимизация порядка

Стартовые **диспетч-правила** (генераторы порядка): `SPT/LPT/EDD/CR/MWKR/MS/FIFO`.
Каждое правило → порядок → симуляция (детерм. + Монте-Карло) → KPI. Выбор лучшего
по целевой функции `score = w_time·E[makespan] + w_risk·(CVaR−E) + просрочка`
(устойчивость к хвосту). Расширение: метаэвристики (ГА/табу/отжиг, локальный поиск
на критическом пути) — порядок как хромосома; оценщик тот же.

## 5. Выходы (три синхронные формы + аналитика)

- **Диаграмма Ганта** — операции по дорожкам станков, цвет по заказу, маркеры
  просрочки.
- **Загрузка оборудования** — занятость/простой каждого станка и типа (% фонда).
- **Планы по рабочим** — операции исполнителя, занятость/простой.
- **Узкое место** — тип оборудования с макс. загрузкой («где теряется время»).
- **Аналитика простоев** — суммарный простой станков и людей.
- **KPI** — makespan, tardiness, OTD, utilization, стоимость; сравнение правил.

## 6. Связь со Стадией 2 (двухстадийность)

Программа Стадии 1 берётся из **портфеля Стадии 2** (`POST /api/schedule {run_id}` →
`portfolio_items`) — «что произвести» → «как произвести». Обратно Стадия 1 даёт
распределения себестоимости/срока/пропускной способности для уточнения портфеля.
Обе стадии **стохастичны** (Стадия 1 — длительности с тяжёлым хвостом; Стадия 2 —
цены) и решаются под устойчивость.

## 7. Контракт API

| Метод | Назначение |
|---|---|
| `POST /api/schedule` | `{program | run_id, rule, w_time/w_cost/w_risk, samples, alpha, tail_weight}` → расписание: Ганта, загрузка, планы рабочих, узкое место, простои, KPI (детерм.+хвост), сравнение правил; **persist** в `plans/plan_tasks`. |

Типы фронтенда — `ScheduleResult` в `src/lib/api.ts`; экран — `features/plan/PlanScreen.tsx`.
