# MAOS — план развития

Документ связывает реализованный фронтенд-срез с требованиями диплома
(онтология + Action layer + планировщик/оптимизатор + сценарии/аналитика +
RBAC + ИИ-ассистент MENTAT) и задаёт ближайшие шаги.

> Полная **прослеживаемость требований** (все документы ТЗ → этапы → код,
> включая модель сущностей, схему БД и окно входа) — в
> [REQUIREMENTS.md](REQUIREMENTS.md). **Архитектурное решение** —
> в [ARCHITECTURE.md](ARCHITECTURE.md). Типы предметной области — `src/domain/`.
>
> **Фундаментальная модель работы** (3D-схема-иерархия подразделение→цех→
> оборудование; справочники-реестры; изделия→техкарты→операции; план→оптимизация
> загрузки оборудования) — каноническая в [CONCEPT.md](CONCEPT.md); разбивка
> интерфейса и все этапы следуют ей.

## Архитектурное решение (принято)

**Фронтенд — веб** (React + TypeScript + three.js, текущий код). **Бэкенд —
на C++** (доменное ядро: онтология, Action layer, планировщик/оптимизатор,
хранение SQLite + ClickHouse, ИИ-агент MENTAT). Связь фронтенд ↔ C++ —
локальный API на `127.0.0.1`; всё работает локально (local-first). Обоснование
и отступление от исходного «C++/Qt desktop» — в [ARCHITECTURE.md](ARCHITECTURE.md).

## Сделано

**3D-схема предприятия** — интерактивная сцена предприятия:

- каркас приложения: Vite + React 18 + TypeScript, дизайн-токены Palantir;
- сцена three.js / R3F: тёмный фон, сетка-блюпринт, тени, орбитальная камера;
- 11 узлов предприятия (3D-модели), раскладка по технологической цепочке
  (демо — металлургический комбинат полного цикла);
- потоки трёх типов: материальные, энергетические, газовые (`FlowKind`/`FLOW_META`);
- выбор узла, всплывающие подписи, инспектор показателей (KPI), статусы;
- данные схемы вынесены в `layout.ts`.

**Оболочка приложения (этап 1)** — рельса навигации, верхняя панель
(с переключателем темы), статус-бар и реестр экранов: Схема, Мониторинг,
План, Оптимизация, Сценарии, Администрирование (последние — заглушки).

**Бэкенд C++ + БД + устойчивая оптимизация** — локальный сервер (SQLite),
все реестры **связаны по ID** (FK + join-таблицы), 3D-схема хранится в БД и
доступна агрегатом `/api/scheme`. Реализована **устойчивая стохастическая
оптимизация**: цены продукции — внешние условия (распределения в сценариях),
Монте-Карло строит распределение прибыли портфелей, выбирается **самое устойчивое**
решение (CVaR/худший случай/mean-variance/min-regret), а не самое прибыльное в
среднем. Экраны «Сценарии», «Оптимизация», «План» — рабочие. Вход — через реестр
пользователей (RBAC), не хардкод-пароль. Миграции по `PRAGMA user_version`
сохраняют существующие данные. Модель — [OPTIMIZATION.md](OPTIMIZATION.md).

**Архитектура и требования** — зафиксировано решение (веб + C++),
заведены типы предметной области `src/domain/*` (НСИ, запасы, планирование,
граф, безопасность) и составлена прослеживаемость всех документов ТЗ
([REQUIREMENTS.md](REQUIREMENTS.md), [ARCHITECTURE.md](ARCHITECTURE.md)).

Проверено: `npm run typecheck` (0 ошибок), `npm run build` (успех).

## Редактор 3D-схемы и иерархия (ключевой пункт плана)

Сейчас схема описывается данными (`layout.ts`) и только просматривается.
Цель — сделать её **создаваемой и редактируемой прямо в 3D**, с иерархией.
Модель данных уже заведена: `src/domain/graph.ts` (`SchemeGraph`, `GraphNode`
с `parentId`/`level`, `GraphEdge` с типом потока).

**Редактор схемы:** палитра узлов, добавление перетаскиванием, перемещение по
сетке, поворот, удаление; рисование связей с выбором типа потока
(материал/энергия/газ/информация); режимы «просмотр»/«редактирование»;
сохранение раскладки через Action layer; валидация.

**Иерархия (drill-down):** узел раскрывается во вложенную подсхему
(цех → участки → операции/станки); навигация по уровням, «хлебные крошки»;
LOD и агрегация KPI снизу вверх.

## Ближайшие шаги по 3D-срезу

1. Редактор схемы: добавление/перемещение/удаление узлов и связей.
2. Иерархия: drill-down во вложенные подсхемы, «хлебные крошки», уровни.
3. Тонкая настройка координат/масштабов узлов и подписей; единая «земля».
4. Error boundary на загрузку моделей; прогресс-индикатор загрузки уже есть.
5. Code-splitting (`lazy` Canvas / three / drei) — бюджет бандла сейчас ~1.1 МБ.
6. Инстансинг + LOD для масштабирования числа узлов и стабильных ~60 FPS.

## Стадии до полноценного продукта (по порядку)

> Очередь по зависимостям: каждая стадия разблокирует следующие. ТЗ — в папке
> «Диплом MAOS». Статус: ✅ есть · 🟡 частично · ⬜ план.

### Стадия A. Производственная программа (заказы) ⬜ ← начинаем
Реестр **заказов** (изделие, кол-во, срок, приоритет, дата запуска) — вход
Стадии 1. Таблица `demand_orders`, CRUD, развёртка в план; источник программы для
расписания (вместо демо). ТЗ: «Требования к производственному плану» §2.
Разблокирует: всё планирование, аналитику, сценарии.

### Стадия B. Расписания и календари ✅
Смены/выходные → фонд рабочего времени; операции **«растягиваются»** через
нерабочие интервалы (calendar-aware планировщик: `nextWork`/`advance`).
Таблицы `schedules`/`shifts`/`calendar_exceptions` + смены по умолчанию (5/2, 06–22);
`/api/calendar` get/put; переключатель «учитывать рабочее время» на экране «План»
(makespan и загрузка считаются по реальному фонду смен). ТЗ: «Требования к
расписаниям и календарям».

### Стадия C. Запасы / MRP ✅
Разузлование программы по цепочке (`operation_inputs`) → потребность в материалах
(gross/net с учётом страх. запаса), сигналы дефицита/дозаказа, срок поставки;
поля `safety_stock`/`reorder_point`/`lead_time_hours` у изделий, `stock_moves`,
начальные остатки сырья; `/api/mrp`; панель MRP на экране «План» (материальная
осуществимость программы). ТЗ: «Требования к запасам». Осталось: материальная
доступность как ограничение в самом планировщике (release-офсет по сроку поставки).

### Стадия D. Слой действий и аудит (Action layer) ✅
Каждое изменение НСИ, прогон оптимизации и расписания фиксируется в журнале
событий (`actions`: actor/entity/type/payload). Экран **«Администрирование»**:
журнал действий (фильтр), пользователи (без хеша), роли и права (RBAC);
`/api/users`, `/api/roles`. ТЗ: «Архитектура_Palantir» (B). Осталось:
статус-воркфлоу действий (PENDING→APPROVED→EXECUTED) и энфорсмент прав на write —
делается вместе со Стадией G (окно входа + RBAC).

### Стадия E. Менеджер сценариев (what-if) 🟡
Единый `scenario` (даты, режим детерм./стох., веса цели) + `scenario_override`
(точечные замены без копии данных); клонирование, сравнение бок о бок. Связывает
цены (Стадия 2) и план (Стадия 1) одним сценарием. ТЗ: «Сценарии моделирования».

### Стадия F. Мониторинг и KPI-аналитика 🟡
ETL KPI (makespan/tardiness/utilization/WIP/OTD/стоимость/риск) в `ts_data`
(опц. ClickHouse); дашборды, тренды, тепловые карты загрузки, критический путь.
ТЗ: «Производственный план» (KPI), «Модель данных».

### Стадия G. Окно входа и RBAC-энфорсмент 🟡
Argon2id, vault (.db)→KDF→SQLCipher, сессия/автоблокировка, защита от перебора;
проверка прав на каждом write-endpoint. ТЗ: «Требования к пользователям, ролям и
окну входа». (Сейчас: вход против `users/roles`, демо-хеш.)

### Стадия H. Импорт/экспорт (ETL) ⬜
Импорт Excel-ТЗ (реестры, техкарты, нормы, программа) с валидацией и предпросмотром;
экспорт плана/Ганта/KPI в XLSX/CSV/PDF. ТЗ: «Импорт и экспорт данных».

### Стадия I. Перепланирование и правки плана ⬜
Drag&drop работ, pin, «станок недоступен», вставка срочного заказа; инкрементальный
пересчёт хвоста, предпросмотр последствий, undo/redo, план-факт. ТЗ:
«Производственный план», «Эвристики».

### Стадия J. Метаэвристики (обе стадии) ⬜
ГА (хромосома = порядок операций), табу, отжиг, локальный поиск на критическом
пути; anytime, Парето-фронт. ТЗ: «Требования эвристики», «…к оптимизации».

### Стадия K. Обратная связь Стадия 1 → Стадия 2 ⬜
Распределения себестоимости/срока/реальной пропускной способности из расписания →
в выбор портфеля (полная двухстадийная стохастическая программа с рекурсом).

### Стадия L. MENTAT — ИИ-ассистент ⬜
Заземлённый помощник: read-only (запросы/объяснение узких мест/сравнение сценариев),
затем write через Action-ы с подтверждением. Локальная модель (Qwen) + инструменты.
ТЗ: «Архитектура_Palantir» (C).

### Сквозные (НФТ)
Производительность (≤1–2 с план), надёжность/целостность, local-first/шифрование,
RU/EN, тесты/CI, подписанные инсталляторы. ТЗ: «Нефункциональные требования».
