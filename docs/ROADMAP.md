# MAOS — план развития

Документ связывает реализованный фронтенд-срез с требованиями диплома
(онтология + Action layer + планировщик/оптимизатор + сценарии/аналитика +
RBAC + ИИ-ассистент MENTAT) и задаёт ближайшие шаги.

## Сделано (текущий срез)

**3D-схема предприятия** — интерактивный «цифровой двойник»:

- каркас приложения: Vite + React 18 + TypeScript, дизайн-токены Palantir;
- сцена three.js / R3F: тёмный фон, сетка-блюпринт, тени, орбитальная камера;
- 11 узлов предприятия (3D-модели), раскладка по технологической цепочке;
- анимированные потоки (материальные/энергетические) между узлами;
- выбор узла, всплывающие подписи, инспектор показателей (KPI), статусы;
- данные схемы вынесены в `layout.ts` (узлы/координаты/потоки/KPI).

Проверено: `npm run typecheck` (0 ошибок), `npm run build` (успех),
дев-сервер отдаёт страницу и 3D-ассеты (HTTP 200).

## Ближайшие шаги по 3D-срезу

1. Тонкая настройка координат/масштабов узлов и подписей; единая «земля».
2. Error boundary на загрузку моделей; прогресс-индикатор загрузки уже есть.
3. Code-splitting (`lazy` Canvas / three / drei) — бюджет бандла сейчас ~1.1 МБ.
4. Инстансинг + LOD для масштабирования числа узлов и стабильных ~60 FPS.
5. Drill-down: клик по участку → вложенная схема операций/оборудования.
6. Адаптив и доступность (фокус, клавиатура, контраст подписей).

## Дорожная карта (фронтенд-трек MAOS)

> Сопоставлено с фазами диплома; здесь — реализация прикладного слоя.

### Этап 1. Оболочка приложения (shell)
Рельса навигации, докуемые панели, статус-бар, командная палитра в стиле
Blueprint. Экраны: **Схема**, **Мониторинг**, **План/Гантт**,
**Оптимизация**, **Сценарии**, **Администрирование**.

### Этап 2. НСИ (справочники) вместо статики
Доменная модель: изделия, операции, техкарты, типы рабочих центров, станки,
рабочие, компетенции, нормы времени, заказы, запасы. Импорт из JSON/Excel-ТЗ
с валидацией. Узлы 3D-схемы связываются со справочниками.

### Этап 3. Состояние и слой действий (Action layer)
Стор (zustand) + типизированные Action-ы с журналом изменений — фундамент
Palantir-слоя: выбор/drill-down, сценарные оверрайды, аудит мутаций.

### Этап 4. Планировщик (симуляция + эвристики)
Списочная/событийная симуляция с ограничениями (предшествование,
двойной ресурс «станок+рабочий», смены/календарь, компетенции, наладки,
материальная доступность). Диспетч-правила (SPT/EDD/CR/…). Представления:
Гантт, загрузка ресурсов, KPI.

### Этап 5. Оптимизация (поиск порядка)
Целевая `F = w_time·ВРЕМЯ + w_cost·СТОИМОСТЬ + w_risk·РИСК`. Метаэвристики
(ГА/табу/отжиг/локальный поиск), anytime-режим, сравнение прогонов, Парето-фронт.

### Этап 6. Сценарии и аналитика
Менеджер сценариев (оверрайды, клонирование, сравнение бок о бок). Временные
ряды KPI (загрузка, запасы, наработка), дашборды, подсветка критического пути
и просрочек на 3D-схеме и Гантте.

### Этап 7. Пользователи, роли, вход (RBAC)
Идентичность (рабочий = пользователь), роли и права, окно входа, проверка прав
на каждом Action-е, аудит входов.

### Этап 8. MENTAT — ИИ-ассистент на данных предприятия
Заземлённый в онтологии диалоговый помощник: сначала read-only (запросы к
объектам/KPI, объяснение узких мест, сравнение сценариев), затем write через
Action-ы с обязательным подтверждением человеком (human-in-the-loop).
Локальная модель (например, Qwen) + локальный RAG по регламентам/ТЗ.

## Архитектурная развилка (требует решения)

Диплом описывает целевую реализацию как **десктоп C++/Qt** (local-first,
SQLite/SQLCipher + ClickHouse, локальная LLM). Текущий код — **веб-фронтенд**
(React/R3F), что позволило переиспользовать готовый 3D-подход и модели.

Варианты:
- **A. Веб-фронтенд + локальный бэкенд** (например, Tauri/Electron-оболочка
  или локальный сервис) — сохраняем 3D-визуализацию и local-first.
- **B. Фронтенд как прототип визуализации**, доменное ядро — отдельно (C++/Qt),
  как в дипломе; 3D-схема портируется в Qt Quick 3D позже.

Рекомендация: зафиксировать развилку до Этапа 2 (НСИ), т.к. от неё зависит
способ хранения данных и интеграции домена.
