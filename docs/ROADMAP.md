# MAOS — план развития

Документ связывает реализованный фронтенд-срез с требованиями диплома
(онтология + Action layer + планировщик/оптимизатор + сценарии/аналитика +
RBAC + ИИ-ассистент MENTAT) и задаёт ближайшие шаги.

> Полная **прослеживаемость требований** (все документы ТЗ → этапы → код,
> включая модель сущностей, схему БД и окно входа) — в
> [REQUIREMENTS.md](REQUIREMENTS.md). **Архитектурное решение** —
> в [ARCHITECTURE.md](ARCHITECTURE.md). Типы предметной области — `src/domain/`.
>
> **Фундаментальная модель работы** (3D-схема-иерархия подразделение→цех→
> оборудование; справочники-реестры; изделия→техкарты→операции; план→оптимизация
> загрузки оборудования) — каноническая в [CONCEPT.md](CONCEPT.md); разбивка
> интерфейса и все этапы следуют ей.

## Архитектурное решение (принято)

**Фронтенд — веб** (React + TypeScript + three.js, текущий код). **Бэкенд —
на C++** (доменное ядро: онтология, Action layer, планировщик/оптимизатор,
хранение SQLite + ClickHouse, ИИ-агент MENTAT). Связь фронтенд ↔ C++ —
локальный API на `127.0.0.1`; всё работает локально (local-first). Обоснование
и отступление от исходного «C++/Qt desktop» — в [ARCHITECTURE.md](ARCHITECTURE.md).

## Сделано

**3D-схема предприятия** — интерактивная сцена предприятия:

- каркас приложения: Vite + React 18 + TypeScript, дизайн-токены Palantir;
- сцена three.js / R3F: тёмный фон, сетка-блюпринт, тени, орбитальная камера;
- 11 узлов предприятия (3D-модели), раскладка по технологической цепочке
  (демо — металлургический комбинат полного цикла);
- потоки трёх типов: материальные, энергетические, газовые (`FlowKind`/`FLOW_META`);
- выбор узла, всплывающие подписи, инспектор показателей (KPI), статусы;
- данные схемы вынесены в `layout.ts`.

**Оболочка приложения (этап 1)** — рельса навигации, верхняя панель
(с переключателем темы), статус-бар и реестр экранов: Схема, Мониторинг,
План, Оптимизация, Сценарии, Администрирование (последние — заглушки).

**Бэкенд C++ + БД + устойчивая оптимизация** — локальный сервер (SQLite),
все реестры **связаны по ID** (FK + join-таблицы), 3D-схема хранится в БД и
доступна агрегатом `/api/scheme`. Реализована **устойчивая стохастическая
оптимизация**: цены продукции — внешние условия (распределения в сценариях),
Монте-Карло строит распределение прибыли портфелей, выбирается **самое устойчивое**
решение (CVaR/худший случай/mean-variance/min-regret), а не самое прибыльное в
среднем. Экраны «Сценарии», «Оптимизация», «План» — рабочие. Вход — через реестр
пользователей (RBAC), не хардкод-пароль. Миграции по `PRAGMA user_version`
сохраняют существующие данные. Модель — [OPTIMIZATION.md](OPTIMIZATION.md).

**Архитектура и требования** — зафиксировано решение (веб + C++),
заведены типы предметной области `src/domain/*` (НСИ, запасы, планирование,
граф, безопасность) и составлена прослеживаемость всех документов ТЗ
([REQUIREMENTS.md](REQUIREMENTS.md), [ARCHITECTURE.md](ARCHITECTURE.md)).

Проверено: `npm run typecheck` (0 ошибок), `npm run build` (успех).

## Редактор 3D-схемы и иерархия (ключевой пункт плана)

Сейчас схема описывается данными (`layout.ts`) и только просматривается.
Цель — сделать её **создаваемой и редактируемой прямо в 3D**, с иерархией.
Модель данных уже заведена: `src/domain/graph.ts` (`SchemeGraph`, `GraphNode`
с `parentId`/`level`, `GraphEdge` с типом потока).

**Редактор схемы:** палитра узлов, добавление перетаскиванием, перемещение по
сетке, поворот, удаление; рисование связей с выбором типа потока
(материал/энергия/газ/информация); режимы «просмотр»/«редактирование»;
сохранение раскладки через Action layer; валидация.

**Иерархия (drill-down):** узел раскрывается во вложенную подсхему
(цех → участки → операции/станки); навигация по уровням, «хлебные крошки»;
LOD и агрегация KPI снизу вверх.

## Ближайшие шаги по 3D-срезу

1. Редактор схемы: добавление/перемещение/удаление узлов и связей.
2. Иерархия: drill-down во вложенные подсхемы, «хлебные крошки», уровни.
3. Тонкая настройка координат/масштабов узлов и подписей; единая «земля».
4. Error boundary на загрузку моделей; прогресс-индикатор загрузки уже есть.
5. Code-splitting (`lazy` Canvas / three / drei) — бюджет бандла сейчас ~1.1 МБ.
6. Инстансинг + LOD для масштабирования числа узлов и стабильных ~60 FPS.

## Стадии до полноценного продукта (по порядку)

> Очередь по зависимостям: каждая стадия разблокирует следующие. ТЗ — в папке
> «Диплом MAOS». Статус: ✅ есть · 🟡 частично · ⬜ план.

### Стадия A. Производственная программа (заказы) ⬜ ← начинаем
Реестр **заказов** (изделие, кол-во, срок, приоритет, дата запуска) — вход
Стадии 1. Таблица `demand_orders`, CRUD, развёртка в план; источник программы для
расписания (вместо демо). ТЗ: «Требования к производственному плану» §2.
Разблокирует: всё планирование, аналитику, сценарии.

### Стадия B. Расписания и календари ✅
Смены/выходные → фонд рабочего времени; операции **«растягиваются»** через
нерабочие интервалы (calendar-aware планировщик: `nextWork`/`advance`).
Таблицы `schedules`/`shifts`/`calendar_exceptions` + смены по умолчанию (5/2, 06–22);
`/api/calendar` get/put; переключатель «учитывать рабочее время» на экране «План»
(makespan и загрузка считаются по реальному фонду смен). ТЗ: «Требования к
расписаниям и календарям».

### Стадия C. Запасы / MRP ✅
Разузлование программы по цепочке (`operation_inputs`) → потребность в материалах
(gross/net с учётом страх. запаса), сигналы дефицита/дозаказа, срок поставки;
поля `safety_stock`/`reorder_point`/`lead_time_hours` у изделий, `stock_moves`,
начальные остатки сырья; `/api/mrp`; панель MRP на экране «План» (материальная
осуществимость программы). ТЗ: «Требования к запасам». Осталось: материальная
доступность как ограничение в самом планировщике (release-офсет по сроку поставки).

### Стадия D. Слой действий и аудит (Action layer) ✅
Каждое изменение НСИ, прогон оптимизации и расписания фиксируется в журнале
событий (`actions`: actor/entity/type/payload). Экран **«Администрирование»**:
журнал действий (фильтр), пользователи (без хеша), роли и права (RBAC);
`/api/users`, `/api/roles`. ТЗ: «Архитектура_Palantir» (B). Осталось:
статус-воркфлоу действий (PENDING→APPROVED→EXECUTED) и энфорсмент прав на write —
делается вместе со Стадией G (окно входа + RBAC).

### Стадия E. Менеджер сценариев (what-if) 🟡 (ядро готово)
Единый `scenario` = цены (распределения + корреляция) **+ цель оптимизации**
(objective/alpha/max_share) — теперь сценарии реально различаются в робастном решении.
**Сделано:** параметры цели в `price_scenarios` (migrate_v11), `/api/optimize` берёт цель
из сценария; **клонирование** (`POST /api/scenarios/:id/clone` — копия параметров и
распределений); экран **«Сравнение сценариев»** — выбор двух, прогон оптимизации каждого,
таблица бок о бок (ожид. прибыль робаст/жадный, CVaR, худший случай, p(убыток), цена
робастности, эфф. число изделий, концентрация, топ-изделия) с подсветкой лучшего.
Сценарий также несёт параметры **прогноза цен** (инфляция/курс/спрос/волатильность/
горизонт) и **режим** детерм.↔стох.: `/api/forecast` принимает `scenario_id` и гоняет
прогноз по сценарию; детерминированный режим схлопывает веер в точечный прогноз. Экран
«Прогноз цен» — выпадающий список сценариев; «Сравнение» — переключатель режима.
**Осталось:** `scenario_override` (точечные замены отдельных значений без копии),
привязка к **плану** (Стадия 1) тем же сценарием, даты периода. ТЗ: «Сценарии».

### Стадия E+. Внешние условия и динамика по времени 🟡 (начато)
Время как первый класс: внешние условия, влияющие на цены, и профили во времени.
1. **Прогноз цен по месяцам** ✅ — `/api/forecast`: коррелированное лог-нормальное
   блуждание цен изделий/сырья на горизонт (мес.) с веером P10/P50/P90; экран
   «Прогноз цен» (графики цены во времени + дельты).
2. **Макрофакторы, двигающие цены** ✅ — инфляция (мес. дрейф), курс (fx),
   индекс спроса; общий рыночный шок задаёт корреляцию. Управляются на экране
   прогноза; накопленный индекс инфляции.
3. **Загрузка оборудования по времени** ✅ — тепловая карта «станок × время» из
   Gantt (24 бина), доля занятости бина → загрузка/простой; вкладка «во времени»
   в блоке загрузки экрана План.
   _Дальше:_ привязка прогноза цен к сценарию/оптимизации (multi-period), сохранение
   траекторий в `ts_data`, FX/спрос по продуктам, сезонность.

### Стадия E++. Историческая БД цен + подбор распределений + EMD/SD ⬜ (требования)
**Контекст:** текущий `/api/forecast` — упрощённый параметрический GBM от одной
`base_price`. Эталонные методики (248.DTE, 133.DtwinLib, CB_models_stohastic)
строят прогноз ОТ ИСТОРИИ рядов и ПОДБИРАЮТ распределения. Чтобы их переиспользовать,
нужны данные и модули ниже. Дорого и data-dependent — выделено явно.

**F0. Историческая БД цен (`ts_data`) — предпосылка всего остального.** ⬜
- Таблица `ts_data(ts_id, product_id, kind[price|cost], date, value)` + индексы; ETL
  загрузки/ввода истории по изделиям и сырью (помесячно, ≥24–36 точек).
- Без истории EMD/SD и подбор распределений не работают (нужен ряд, не точка).
- Аналог: ClickHouse `TS_DATA`/`TS_INFO` в 248.DTE.

**F1. Подбор распределений (reuse «как подбирались»).** 🟡
- Сделано: кандидаты **нормаль / Лаплас / t-Стьюдент** (ν из эксцесса) с выбором по **AIC**
  на лог-доходностях истории; t-сэмплинг как мультивариативный-t (общий гауссов фактор /
  √(χ²_ν/ν)) — сохраняет корреляцию. Осталось: **Парето** и **α-stable** (нужна численная
  плотность `dstable`/характеристическая функция — тяжёлый порт).
- Порт из `CB_models_stohastic`: фит кандидатов по MLE и **выбор по AIC**
  (`CalibrateStochasticDistributions`, `EstimateHeavyTailParams`, `estimate_stable_params`,
  `all_fun.R:2587–2788`). Кандидаты: нормаль, логнормаль, gamma, t-Стьюдент, Парето,
  **α-stable** (тяжёлые хвосты). Критерий: AIC/BIC (+ опц. KS/AD GoF).
- Применять к **лог-доходностям** исторического ряда цены каждого изделия/сырья →
  параметры распределения шока для Монте-Карло. C++: нужен оптимизатор (NLopt/Eigen),
  α-stable плотность через характеристическую функцию.

**F2. Прогноз от истории (reuse «как делал прогноз»).** ⬜
- GBM с μ,σ ИЗ ИСТОРИИ: `μ = mean(diff(log y))`, `σ` из реализованной волатильности
  (`133.DtwinLib/R/EMD_SD.R:194–208`); хвостовой шок — из подобранного распределения (F1),
  а не из фиксированной σ.
- **EMD+SD спектральная декомпозиция** (детерминированный тренд+гармоники):
  `133.DtwinLib/R/sd_fun.R` (Лаплас-преобразование + 13 осцилляторов, матричный solve)
  и `EMD_SD.R` (EMD `emd()` + GBM-остаток + HAR-волатильность). C++: Eigen для SVD/solve,
  EMD — сложно (вариант: FFT-разложение).
- Инфляция как **кумулятивный индекс** `cumprod(1+iₘ)` из помесячного ряда (248.DTE
  `14.Prices_forecast.R:get_data_prep_price`), а не одна ставка.

**F3. Зависимость рядов — копулы.** ⬜
- Заменить однофакторную гауссову корреляцию на **копульную** (Clayton/Gumbel/Frank/Joe,
  хвостовая зависимость), `13.copulas_exsample.R` (`fitCopula`/`rCopula`). C++: ковариация
  + Cholesky как минимум; копулы — отдельный модуль.

**КАНОН (зафиксировано): единый движок прогноза для СТАВКИ и ЦЕН.**
Способ, которым в `CB_models_stohastic` прогнозируется ключевая ставка ЦБ, принимается
КАНОНИЧЕСКИМ и применяется к ЕДИНОМУ набору прогнозируемых рядов — **ключевая ставка
И цены изделий/сырья** прогнозируются одним и тем же стохастическим движком:
1. оценка параметров из истории (μ из лог-доходностей, σ из реализованной волатильности,
   HAR; `Main.R:326–361`);
2. **подбор распределения шока по AIC** (нормаль → Лаплас → t-Стьюдент → Парето →
   **α-stable** тяжёлые хвосты; `all_fun.R:2587–2788`, CMS-сэмплер `FastMonteCarlo.cpp`);
3. **режимы HMM** (7 состояний, Баум–Уэлч/Витерби; `Main.R:366–438`) — переключение
   режимов рынка/ДКП;
4. **SVD-VAR** прогноз макродрайверов (`all_fun.R:44–275`);
5. **Монте-Карло** (10k путей) → веер квантилей P5..P95.
Ключевая ставка — **внешнее условие верхнего уровня**: через правило Тейлора и инфляцию
она задаёт дрейф цен (связь ставка→цена — на шаге F2/Стадия J). Таким образом «и сама
ставка, и цены» считаются единообразно; ряд ставки лежит в той же `ts_data`/`macro_ts`.

**Порядок:** F0 (данные) → F1 (подбор распределений) → F2 (прогноз от истории, ставка→цена)
→ F3 (копулы). Сделано: F0 (таблица `ts_data` + сид истории), F1-старт (data-driven μ/σ +
выбор нормаль↔Лаплас по AIC в `/api/forecast`), прогноз **ключевой ставки** тем же движком
(`macro_ts`). Дальше: α-stable/t/Парето кандидаты, HMM-режимы, SVD-VAR, EMD/SD, копулы —
крупный C++-порт (~6–8 недель по разбору эталонов), отдельными сессиями.
