# MAOS — архитектурное решение

> Статус: **принято** (2026-06-26). Этот документ фиксирует ключевое
> архитектурное решение и его последствия. Прослеживаемость требований —
> в [REQUIREMENTS.md](REQUIREMENTS.md). Фундаментальная модель работы —
> в [CONCEPT.md](CONCEPT.md).

## Решение (ADR-001): веб-фронтенд + бэкенд на C++

- **Фронтенд — веб.** React + TypeScript + three.js (@react-three/fiber/drei).
  Это текущий код в репозитории: 3D-схема предприятия, оболочка приложения,
  экраны. Веб оставляем как есть.
- **Бэкенд — на C++.** Доменное ядро и тяжёлая логика реализуются на C++:
  - онтология (object/link types) и **Action layer** (event sourcing, аудит);
  - **планировщик** (симуляция-оценщик) и **оптимизатор** (эвристики/метаэвристики,
    опц. CP-SAT) — там, где нужен параллелизм и скорость (C++);
  - **хранение**: OLTP — **SQLite** (WAL, FOREIGN KEY, опц. SQLCipher),
    OLAP — **ClickHouse** (временные ряды KPI/наработки/запасов);
  - **MENTAT** — локальный ИИ-агент (read → write с подтверждением).
- **Связь фронтенд ↔ бэкенд** — локальный API на `127.0.0.1`
  (HTTP + WebSocket для стрима прогресса оптимизации/событий). Сериализация —
  JSON (DTO). Никаких внешних сервисов в рантайме — **local-first**.
- **Поставка** — десктоп-оболочка (Tauri/Electron) либо локальный запуск:
  C++-сервис стартует рядом с UI, данные не покидают машину.

## Контур системы

```
┌──────────────────────── Десктоп (local-first) ────────────────────────┐
│                                                                        │
│  ВЕБ-ФРОНТЕНД (этот репозиторий)            БЭКЕНД на C++               │
│  React + TS + three.js / R3F                Доменное ядро (без UI)      │
│  ┌──────────────────────────┐               ┌───────────────────────┐  │
│  │ Оболочка (rail/top/status)│   HTTP/WS    │ Онтология + Action layer │ │
│  │ Схема 3D (узлы, потоки)   │◄────────────►│ Планировщик (симуляция)  │ │
│  │ Редактор схемы (иерархия) │  127.0.0.1   │ Оптимизатор (эвр./мета)  │ │
│  │ План/Гантт, KPI, Сценарии │   JSON DTO   │ MENTAT (LLM, локально)   │ │
│  │ Окно входа (vault/логин)  │              │ Доступ/RBAC, аудит        │ │
│  └──────────────────────────┘               └───────────┬───────────┘  │
│        ▲ контракт типов: src/domain/*.ts                 │              │
│                                              ┌───────────┴───────────┐  │
│                                              │ SQLite (OLTP, .db)    │  │
│                                              │ ClickHouse (OLAP)     │  │
│                                              └───────────────────────┘  │
└────────────────────────────────────────────────────────────────────────┘
```

## Граница фронтенд / бэкенд

| Слой | Где | Технологии |
|---|---|---|
| Визуализация 3D, экраны, редактор схемы, инспектор, Гантт | Фронтенд | React, three.js/R3F |
| Типы предметной области (контракт) | Фронтенд | `src/domain/*.ts` (зеркало доменных сущностей) |
| Онтология, Action layer, аудит | Бэкенд | C++ |
| Планировщик (симуляция), оптимизатор, эвристики | Бэкенд | C++ (+ параллелизм, опц. OR-Tools) |
| Хранение OLTP / OLAP | Бэкенд | SQLite (+SQLCipher) / ClickHouse |
| MENTAT (агент), RBAC, импорт ТЗ | Бэкенд | C++ (llama.cpp/Ollama локально) |

`src/domain/*.ts` — это **типизированный контракт** фронтенда с C++-ядром
(DTO/вью-модели). Он повторяет доменные сущности из требований (см.
[REQUIREMENTS.md](REQUIREMENTS.md)) и служит точкой синхронизации.

## Обоснование

- **Переиспользование 3D.** Готовый подход и модели (three.js/R3F) переносятся
  в веб без переписывания на Qt Quick 3D.
- **C++ для алгоритмов.** Планировщик/оптимизатор и работа с большими временными
  рядами выигрывают от C++ (скорость, параллелизм) — как и требует ТЗ
  (≤ 1–2 с на типовой задаче; CP-SAT в бюджете времени).
- **Чёткая граница.** UI и домен развязаны через локальный API; домен не зависит
  от UI (чистая архитектура из «Нефункциональных требований»).

## Отступление от исходного ТЗ (осознанное)

«Нефункциональные требования» и «План реализации» описывали **единый
C++/Qt desktop** с 3D через Qt RHI. Принятое решение **меняет UI-слой на веб**
(React/R3F), сохраняя C++ для домена/алгоритмов/хранения/агента. Последствия:

- кроссплатформенность UI обеспечивается вебом (+ десктоп-оболочка), а не Qt;
- 3D-рендер — WebGL (three.js), а не Qt RHI;
- требования к домену, БД, планировщику, RBAC, MENTAT **остаются в силе** и
  реализуются на C++ согласно ТЗ;
- local-first и офлайн-работа сохраняются (API только на `127.0.0.1`).

## Последствия и открытые вопросы

- Нужен слой DTO и версионирование контракта фронтенд ↔ C++.
- Выбор оболочки поставки (Tauri vs Electron vs системный браузер) — отдельное
  решение перед этапом поставки.
- SQLCipher-шифрование и окно входа (vault) — на стороне C++; фронтенд только
  показывает форму и шлёт логин/пароль на локальный бэкенд.
