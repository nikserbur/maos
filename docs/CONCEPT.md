# MAOS — фундаментальная модель работы

> Это **каноническое** описание того, как система работает. Все главные
> документы (README, ROADMAP, REQUIREMENTS, ARCHITECTURE) и **разбивка
> интерфейса** следуют этой модели. При расхождении — приоритет у этого файла.

## 1. 3D-схема предприятия (иерархия)

Предприятие представлено **иерархической 3D-схемой**:

```
Предприятие → Подразделения → Цеха → Оборудование
```

На схеме отображается **оборудование**; у каждой единицы оборудования есть
привязанный **тип оборудования**. Навигация по уровням — drill-down
(вход внутрь узла → вложенная подсхема, «хлебные крошки» для возврата).

## 2. Тип оборудования (справочник, задаётся при внедрении)

**Тип оборудования** — справочник, который **определяется при внедрении**
системы на конкретное предприятие. Тип задаёт **набор параметров** единицы
оборудования. Каждая единица оборудования на схеме имеет свой тип; **параметры
оборудования зависят от типа** и вводятся прямо на схеме / в реестре. Доменная
модель и все необходимые поля оборудования заданы (`src/domain/nsi.ts`).

## 3. Справочники = реестры (НСИ)

Все справочники — это **реестры**: поиск, поля, **аккуратная таблица**,
возможность **заводить/создавать новые** записи прямо в приложении. Реестры:

- **Типы оборудования** — классы взаимозаменяемого оборудования.
- **Оборудование** — единицы с привязкой к типу, подразделению, расписанию.
- **Изделия** — номенклатура с составом (BOM).
- **Техкарты** — маршруты изготовления изделий.
- **Операции** — технологические операции.
- **Рабочие** — персонал и компетенции (тоже ресурс плана).

## 4. Изделия (состав + запасы)

**Изделие** — это нечто, в состав чего входят **другие изделия** (иерархия,
BOM) в нужном **количестве**. У изделий есть **запасы** — отдельно, как на
складе (склад — это локация/тип оборудования). В реестре изделий записи можно
заводить. Изделие чётко **привязано к техкартам** — как именно его произвести;
для разных изделий — разные операции на разном оборудовании.

## 5. Техкарты (последовательность операций)

**Техкарта** изделия = **последовательность технологических операций**, которые
нужно выполнить, чтобы произвести изделие. В приложении: **реестр техкарт**
(поиск, таблица), создание техкарты = **задание последовательности операций**.

## 6. Операции

**Операция** — последовательность действий, у которой есть **затраты**:
**время, ресурсы, риск** и **стоимость**. Операция привязана:

- к **конкретному изделию** (через техкарту изделия),
- к **типу оборудования**, на котором её можно выполнять,
- имеет **порядок** в маршруте (последовательность),
- может требовать **набор входных изделий** для выполнения (если нужен).

В приложении — **реестр операций** (поиск, таблица, создание).

## 7. Производственный план и оптимизация

1. Формируем **производственную программу**: выбираем **изделия** + **дату**
   (к какому сроку выпустить).
2. **Оптимизируем производство** — то есть **оптимально загружаем оборудование**
   (и рабочих). Цель: оптимальная загрузка ресурсов → выше **пропускная
   способность**, ниже время/издержки/риск. В этом смысл оптимизации.
3. Рабочие — тоже ограниченный ресурс плана (совместно «оборудование + рабочий»).

## 7а. Внешние условия, портфель и устойчивость (стохастика)

Оптимизация ищет **не самое прибыльное, а самое устойчивое к внешним условиям**
решение. **Внешнее условие** — то, чем предприятие не управляет; первичное —
**цена реализации продукции**. Цена не детерминирована: в **сценарии** она задаётся
**стохастическим распределением**.

1. **Сценарий** задаёт распределения цен на товарные изделия (внешние условия).
2. **Себестоимость** каждого изделия собирается снизу вверх по связанным реестрам
   (BOM + техкарты + операции + типы оборудования) — поэтому реестры **связаны по ID**.
3. **Монте-Карло**: сэмплируются цены (и времена/риски операций) → для каждого
   **портфеля продукции** (что и сколько выпускать) строится **распределение прибыли**.
4. Выбирается **устойчивый портфель** по робастному критерию (CVaR / худший случай /
   mean-variance / min-regret), а не по матожиданию. Из него следует **план
   производства** (загрузка оборудования).

Полная модель и формулы — [OPTIMIZATION.md](OPTIMIZATION.md).

## 8. Как этому следует интерфейс (разбивка экранов)

Навигация приложения **повторяет эту модель**:

| Экран | Содержание |
|---|---|
| **Схема** | 3D-схема предприятия (подразделение → цех → оборудование), drill-down |
| **Справочники (НСИ)** | реестры: типы оборудования, оборудование, изделия, техкарты, операции, рабочие — поиск, таблицы, создание |
| **План** | производственная программа = устойчивый портфель + загрузка оборудования |
| **Оптимизация** | устойчивое решение под стохастику цен (робастный vs ожидаемо-лучший, распределение прибыли) |
| **Сценарии** | внешние условия: распределения цен на продукцию (стохастика) |
| **Администрирование** | пользователи, роли, доступ |

## 9. Связь сущностей (одним взглядом)

```
ТипОборудования ──< Оборудование ──(на схеме, в цехе/подразделении)
       ▲                                   
   (операция допустима на типе)            
Изделие ──< Техкарта ──< Операция (порядок; время/стоимость/риск; входные изделия)
   │                                        
   └──< Запас (склад)                       
ПроизводственнаяПрограмма (изделия + дата) → Оптимизация (загрузка оборудования + рабочие) → План
```
