# MAOS — устойчивая стохастическая оптимизация производства

> Статус: **принято** (2026-06-28). Каноническое описание модели планирования и
> оптимизации производства под **неопределённость внешних условий**. Дополняет
> фундаментальную модель ([CONCEPT.md](CONCEPT.md)) разделом «План → Оптимизация»
> и фиксирует ключевое решение **ADR-002** ([ARCHITECTURE.md](ARCHITECTURE.md)).
> Реализация — C++-ядро (`backend/src/Optimizer.*`), контракт — `src/domain/`.

## 0. Главный принцип

Мы ищем **не самое прибыльное, а самое устойчивое** решение. «Лучшее в среднем»
решение часто хрупко: при неблагоприятном стечении внешних условий (падение цен,
рост норм времени, брак) оно даёт убыток. Устойчивое решение сознательно жертвует
частью ожидаемой прибыли ради того, чтобы **держаться в худших сценариях**
(«цена устойчивости», price of robustness).

```
Внешние условия (цены) → стохастические сценарии → Монте-Карло →
  распределение прибыли для каждого портфеля →
    выбор по робастному критерию (CVaR / worst-case / min-regret),
      а не по матожиданию
```

## 1. Что есть «внешние условия»

**Внешнее условие** — то, чем предприятие не управляет, но что определяет
результат. В MAOS первичное внешнее условие — **цена реализации продукции**
(`price`). Цена не детерминирована: она задаётся в **сценарии** как
**стохастическое распределение** на горизонте планирования.

Дополнительно стохастическими (риски при моделировании) считаются:

| Источник неопределённости | Где задаётся | Распределение |
|---|---|---|
| **Цена продукции** `π_p` | сценарий → `price_distributions` | normal / lognormal / triangular / uniform |
| **Время операции** `t_o` | НСИ-операция `t_opt/t_norm/t_pess` | PERT-треугольное (β-PERT) |
| **Риск операции** `r_o` | НСИ-операция `risk_coef` | Бернулли(r) → множитель брака/переделки |
| **Цена закупки сырья** `b_p` | изделие `base_cost` (± вариация) | normal (опц.) |
| **Спрос** `D_p` | изделие `demand_max` (± вариация) | усечённое normal (опц.) |

Каждый **прогон** Монте-Карло — это вектор реализаций `ω = (π, t, r, b, D)`.

## 2. Себестоимость (cost model) — снизу вверх

Себестоимость единицы изделия `p` собирается рекурсивно из связанных реестров —
именно поэтому **все реестры должны быть связаны** (см. §6 и REQUIREMENTS §1):

```
c_p(ω) = Σ_{k ∈ BOM(p)} qty_{k}·c_k(ω)            (материалы/полуфабрикаты, BOM)
       + Σ_{o ∈ routing(p)} opCost_o(ω)            (операции техкарты)
       + b_p(ω)·[p — покупное]                     (закупка сырья)

opCost_o(ω) = t_o(ω)·(machineRate_{wct} + laborRate)
            + setupCost_o·[нужна наладка]
            + cost_o
            ,  с инфляцией на риск:  t_o,opCost ×= (1 + r_o(ω)·reworkFactor)
```

- `BOM(p)` — дочерние изделия (`products.parent_id`, `qty_in_parent`).
- `routing(p)` — операции техкарты изделия (`routings.product_id` → `operations`).
- `wct` — допустимый **тип оборудования** операции (`operation_wc_types` — связь по
  ID, не по имени), его часовая ставка `machineRate`.
- Рекурсия по BOM кэшируется (мемоизация), защита от циклов.

Под стохастикой `c_p` — само **распределение** (т.к. `t_o`, `r_o`, `b_p` случайны).

## 3. Решение: производственный портфель

**Портфель** `x = (x_p)` — план выпуска: сколько единиц каждого **продаваемого**
изделия (`products.sellable = 1`) произвести на горизонте. Ограничения:

- **Мощности**: суммарная загрузка каждого типа оборудования не превышает
  доступный фонд времени `Cap_{wct} = (Σ машин типа wct)·H·efficiency`, где `H` —
  часы горизонта. Потребление: `Σ_p x_p·Σ_{o∈routing(p), wct} t̄_o ≤ Cap_{wct}`.
- **Спрос/сбыт**: `0 ≤ x_p ≤ D_p` (верхняя граница — `demand_max`).
- Целочисленность по партиям (`batch_size`) — округление вниз.

Портфель → **план производства**: загрузка по типам оборудования и список работ
(операция × изделие × объём) — это и есть «планирование производства из
себестоимости внешних условий».

## 4. Реализации цели и распределение прибыли

Для прогона `ω` и портфеля `x`:

```
Profit(x, ω) = Σ_p x_p·( π_p(ω) − c_p(ω) )            (выручка − себестоимость)
             − Penalty(x, ω)                          (штраф за нарушение мощности)
```

По `N` прогонам (общие случайные числа — *common random numbers* — для честного
сравнения портфелей) получаем **эмпирическое распределение прибыли** `{Profit(x, ω_i)}`.

## 5. Робастные критерии — сердце системы

Из распределения `{P_i = Profit(x, ω_i)}` (отсортированного по возрастанию)
считаем метрики:

| Метрика | Формула | Смысл |
|---|---|---|
| **Ожидаемая прибыль** `E` | `mean(P_i)` | «наивно лучшее» — то, что **не** оптимизируем в одиночку |
| **СКО** `σ` | `std(P_i)` | разброс (волатильность результата) |
| **Худший случай** | `min P_i` | критерий Вальда (maximin) |
| **VaR_α** | α-квантиль `P` | порог, ниже которого падаем с вероятностью α |
| **CVaR_α** | `mean(P_i : P_i ≤ VaR_α)` | средние потери в худшие α·100% случаев — **критерий по умолчанию** |
| **P(loss)** | `#{P_i < 0}/N` | вероятность убытка |
| **Downside σ** | СКО только отрицательных отклонений | «плохая» волатильность |
| **Max regret** | `max_i (P*_i − P_i(x))` | макс. сожаление против пооборотного оптимума `P*_i` |

**Выбор устойчивого решения** среди кандидатов `x`:

```
x*_robust   = argmax_x  RobustScore(x)        // по умолчанию RobustScore = CVaR_α
x_expected  = argmax_x  E[Profit(x)]          // «наивно лучшее» — для сравнения
```

Робастный критерий **конфигурируется** (`objective`): `cvar` (по умолчанию) ·
`worstcase` · `meanvariance` (E − λ·σ) · `minregret`. UI всегда показывает
**обе** точки (`x*_robust` и `x_expected`) и «цену устойчивости»
`E[x_expected] − E[x*_robust]`, чтобы решение было прозрачным.

## 6. Почему нужна полная связность реестров

Себестоимость (§2) и мощности (§3) вычислимы, только если связи между реестрами
**настоящие (по ID, с FOREIGN KEY)**, а не условные (по имени/тексту):

```
work_center_type ──< machine            (machine.wc_type_id → FK)  ⇒ фонд мощности
operation ──< operation_wc_types >── work_center_type             ⇒ ставка/мощность
operation ──< operation_inputs   >── product                       ⇒ материалы операции
product (BOM ↺ parent_id, qty) ──< routing ──< operation           ⇒ рекурсия себестоимости
product.sellable, base_cost, demand_max                            ⇒ выручка/спрос/закупка
price_scenario ──< price_distribution >── product                  ⇒ внешние условия
optimization_run ──< portfolio ──< portfolio_item >── product      ⇒ результат
optimization_run ──< plan ──< plan_task >── (operation, wct)        ⇒ план/загрузка
```

Ранее `operation.wc_types` и `operation.inputs` хранились **строками имён** —
переименование типа рвало связь, а оптимизатор не мог собрать модель. Теперь это
**нормализованные join-таблицы по ID**; старые данные **мигрированы** (имена/коды →
ID) без потери.

## 7. Алгоритм (anytime, в бюджете времени/прогонов)

```
1. Загрузить НСИ (изделия, BOM, техкарты, операции, типы, машины) и сценарий.
2. Предрасчёт: список продаваемых изделий, фонд мощности по типам, базовая
   себестоимость (детерминированные средние) для генерации кандидатов.
3. Сгенерировать пул кандидатов-портфелей стратегиями:
   – max ожидаемой маржи · max худшей маржи · диверсификация (равные доли) ·
     пропорционально спросу · жадное заполнение мощностей · случайные пертурбации.
4. Зафиксировать N прогонов (общие случайные числа): сэмплировать π, t, r, b, D.
5. Для каждого кандидата по всем прогонам посчитать Profit → метрики §5.
6. Локальный поиск вокруг лучших по RobustScore (перенос объёма между изделиями).
7. Вернуть: x*_robust, x_expected, ранжированный список кандидатов, гистограмму
   прибыли робастного решения, загрузку ресурсов, разбивку по изделиям. Persist.
```

Сложность контролируется бюджетом (`samples`, число кандидатов, время). Метод
объяснимый и воспроизводимый (`seed`) — требование диплома к сценариям.

## 8. Связь с этапами плана

Закрывает ROADMAP **Этап 5 (планировщик)**, **Этап 6 (оптимизация)** и **Этап 7
(сценарии/аналитика)** в части стохастики и устойчивости:

- Этап 5 — план/загрузка ресурсов из портфеля (§3).
- Этап 6 — целевая функция и робастный выбор (§4–5), anytime (§7).
- Этап 7 — сценарии внешних условий и их сравнение (§1, §5; экран «Сценарии»).

## 9. Контракт API

| Метод | Назначение |
|---|---|
| `GET/POST /api/scenarios` | сценарии внешних условий (+ вложенные `price_distributions`) |
| `GET /api/scenarios/:id` | сценарий с распределениями |
| `POST /api/optimize` | запуск: `{scenario_id, objective, samples, alpha, lambda, seed, horizon_hours}` → прогон с метриками, портфелями, гистограммой; **persist** в `optimization_runs` |
| `GET /api/optimize/runs`, `/:id` | история прогонов и результат |
| `GET /api/scheme` | 3D-схема как единый сохранённый агрегат (узлы+связи+мета) |

Типы фронтенда — `src/domain/optimization.ts` (зеркало DTO). Экран «Оптимизация»
визуализирует распределение и сравнение робастного vs ожидаемо-лучшего.
